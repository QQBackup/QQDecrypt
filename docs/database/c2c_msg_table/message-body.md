# `40800` Protobuf 消息体

`40800` 是 `c2c_msg_table` 中最核心的字段，保存文本、图片、文件、语音、短视频、合并转发、引用和通话等实际消息内容。目标仓库的完整字段报告见 [`40800.md`](https://github.com/QQBackup/ntqq_msg_db_util/blob/master/db_docs/c2c_msg_table/40800.md)。

## 外层结构

可以将外层结构抽象为：

```protobuf
message MsgBody {
  repeated MsgContent content = 40800;
}
```

因此，一行数据库记录不一定只有一个内容段。常见的图文消息通常由多个 `MsgContent` 组成，例如图片后接文字、名片后接附言或位置后接说明。

目标研究样本中，约 124 万行是单段消息，约 7.8 万行是多段消息；多段消息最多可以包含上百个段。多段消息并不等于解码失败，`list` 中的每个元素通常都是一个已经解析出的 `MsgContent`。

## `MsgContent` 的功能分区

| 内容区 | 字段范围 | 主要适用类型 |
| --- | --- | --- |
| 通用头部 | `45001`–`45004` | 所有消息 |
| 文本 | `45101`–`45112` | `40011=2` |
| 媒体基础信息 | `45402`–`45419` | 图片、文件、视频、名片等 |
| CDN 元数据 | `45501`–`45550` | 媒体消息 |
| 图片/视频资源 | `45800`–`45829` | 图片和视频 |
| 短视频位置 tag | `45851`–`45863` | `40011=7` |
| 名片 | `45906`–`45926` | `40011=6` |
| 名片/引用 | `47501`–`47502`、`47702`–`47715` | `40011=5` |
| 合并转发 | `47901`–`47904` | `40011=11` |
| 通话通知 | `48151`–`48157` | `40011=19` |
| 视频/音视频流 | `48401`–`48421` | `40011=10` |
| 旧协议合并转发 | `48601`–`48603` | `40011=8` |
| 通用扩展 | `49154`–`49155` | 多种消息类型 |
| 系统通知 | `80810`–`80995` | `40011=17` |

## 通用头部字段

| 字段 | 含义 |
| --- | --- |
| `45001` | 内容段 ID，与消息主键空间相关；同一消息的多个段通常使用连续 ID |
| `45002` | 内容子类型；常见 `1`=文本，`2`=图片/媒体 |
| `45003` | 媒体子标志，具体语义仍有部分推测 |
| `45004` | 少量空 sub-message，语义未知 |

## 文本内容

当 `40011=2` 且 `45002=1` 时，`45101` 通常是 UTF-8 文本正文。处理时应优先按 bytes 解码，不要因为某些字节恰好能被通用 Protobuf 工具识别成结构就把它当成嵌套对象。

| 字段 | 含义 |
| --- | --- |
| `45101` | 文本正文 |
| `45102` | 文本属性标志，常见为 `0` |
| `45112` | 少量编辑/已读时间信息，含 `50202` 时间戳 |

## 媒体基础字段

| 字段 | 含义 |
| --- | --- |
| `45402` | 文件名，常见为 MD5 字符串加扩展名 |
| `45403` | 本地发送路径 |
| `45405` | 文件大小，单位为字节 |
| `45406` | 16 字节原始 MD5，不是 UTF-8 字符串 |
| `45408` / `45409` | 缩略图或校验 hash，具体用途部分未确认 |
| `45411` / `45412` | 图片宽度和高度 |
| `45419` | 文件扩展名 |

### CDN 与本地缓存

| 字段 | 含义 |
| --- | --- |
| `45503` | CDN 下载 token |
| `45504` | 旧版 CDN URL 路径 |
| `45505` | CDN 过期时间戳 |
| `45510` | 新版 CDN 结构中的第二 token；旧 Protobuf 定义可能错误地将其声明为 sub-message |
| `45802`–`45804` | CDN 下载 URL 路径 |
| `45812` | 本地缓存路径 |
| `45815` | 文本降级标记，例如 `[动画表情]` |
| `45816` | CDN 主机名 |

## 特殊消息类型

### 引用回复

`47710` 可以递归嵌套一个完整的 `MsgContent`，因此引用消息不只是文本摘要；解析器需要允许递归读取同一套字段。

### 合并转发

- `40011=11`：较新的 NT 合并转发，`47901` 通常是 JSON 元数据，`47904` 是转发包 UUID。
- `40011=8`：旧协议合并转发，`48601` 是下载 token，`48602` 是 XML 描述，`48603` 是转发包 UUID。

### 通话通知

`48151` 是通话类型，`48152` 是通话时长（毫秒），`48153` 是可读的通话说明文本。外层 `40012` 通常直接镜像 `48151`。

### 协议版本与内层时间

| 字段 | 含义 |
| --- | --- |
| `49154` | NT 客户端协议版本标识；目标样本中主要出现 `"1"` 和 `"nt_1"` |
| `49155` | 消息内层 Unix 秒级时间戳，通常与外层 `40050` 接近 |

## bytes 字段不要统一当成字符串

目标研究显示，`45406`、`45408`、`45409`、`45407`、`45421` 和 `45862` 等字段包含原始二进制 hash，不能直接按 UTF-8 解码。另一方面，`45101`、`45402`、`45419`、`45503`、`45504`、`45802`–`45804` 等字段多数可以按文本处理。

实际解析时应根据字段语义和取值样本决定解码方式，而不是仅由 Protobuf 的 `bytes` 类型决定。
